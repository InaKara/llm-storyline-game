# Review Findings: Scenario Independence Plan

Reviewed document: [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md)

## Findings

1. High: The plan does not include a scenario-scoped prompt loading and reload path, so runtime-generated scenarios still cannot supply their own responder or narrator prompts. [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L410) says prompt assets will live under scenario-specific paths, and [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L560) treats the generator as a late implementation step, but the current runtime only loads prompts once from [backend/app/prompts](backend/app/prompts) via [backend/app/services/prompt_loader.py](backend/app/services/prompt_loader.py#L12), [backend/app/services/prompt_loader.py](backend/app/services/prompt_loader.py#L29), and [backend/app/services/prompt_loader.py](backend/app/services/prompt_loader.py#L50). That startup load is wired in [backend/app/main.py](backend/app/main.py#L47), while [backend/app/services/game_service.py](backend/app/services/game_service.py#L313) still expects fixed narrator template keys such as archive_discovery and scene_transition. As written, a newly generated scenario can write JSON and images to disk and still run against stale global prompts or require a restart.

2. High: The document misses the public scenario-selection contract, so it does not actually deliver loading scenarios at runtime from the client side. The lower layers already accept a scenario identifier in [backend/app/services/scenario_loader.py](backend/app/services/scenario_loader.py#L30) and [backend/app/services/session_initializer.py](backend/app/services/session_initializer.py#L27), but the public create-session route is still pinned to manor in [backend/app/api/routes.py](backend/app/api/routes.py#L75). The API section of the plan only discusses generic flags and counters in [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L136) and [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L566). Without an explicit change to the session creation request, plus a way to list or validate generated scenarios, the main user-facing runtime-loading goal remains blocked even after the internal schema refactor.

3. Medium: The inventory and rollout order omit the initialization and reset path, which means the proposed first refactor slice would leave the app broken in the middle of the migration. [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L548) and [docs/imp_scenario_independence.md](docs/imp_scenario_independence.md#L560) call out model changes, state updating, game service changes, and API changes, but they never include SessionInitializer. That is a gap because [backend/app/services/session_initializer.py](backend/app/services/session_initializer.py#L39) still materializes archive_unlocked, steward_pressure, StewardState, and HeirState directly, and [backend/app/services/game_service.py](backend/app/services/game_service.py#L179) duplicates the same manor-specific reconstruction during reset. Those files need to be in the first-wave migration plan, not treated as follow-on cleanup.

## Assumptions

- These findings assume runtime loading means a newly generated scenario should be selectable and playable without a server restart.
- If the intent is to keep one shared global prompt set for all scenarios, the plan should state that explicitly, because the current document implies per-scenario prompt assets.
- The validator layer appears mostly generic already in [backend/app/core/validators.py](backend/app/core/validators.py#L8), so I did not treat it as a critical omission.
